Executable tutorial

Run an executable prompt with bound tools.

A compiled prompt can be more than text to paste elsewhere. This tutorial uses the checked-in recurring-topic-monitor to show how WeaveMark declares tools, binds them to trusted Python functions, runs the model with those tools, and saves both the final digest and the execution trace.

You inspect @execute single-call, @tool, and @bind.
You run A source-grounded monitor that searches, crawls, and synthesizes.
You produce A monitor digest plus an auditable tool-call trace.

0. Setup

Work from the repository root. The runner uses the installed weavemark command and the configured model/search environment. Because this example calls live web-search and crawl tools, it may consume model and provider quota. 

cd weavemark
pip install -e ".[dev]"
weavemark --help
Trust boundary: executable prompts can call code. Run checked-in bindings you understand, and review recurring_topic_monitor.py before allowing the promplet to use it.

1. Read the executable header

The monitor starts by declaring that it is executable. Unlike a prompt-only spec, this one can run a model with tool calls inside the WeaveMark runtime. 

@promplet version: 0.7

@execute single-call
  max_iterations: 20
  max_tool_calls: 30

@refine module:weavemark.std.reasoning.base_analyst mingle: true
@refine module:weavemark.std.guidelines.evidence_quality mingle: true
@refine module:weavemark.domains.research.recurring_topic_monitor mingle: true

@execute single-call means the runtime gives one model conversation access to the declared tools until it finishes or hits the explicit iteration and tool-call caps. The caps are part of the safety and cost contract.

2. Bind Python functions as tools

Tool names in the promplet are not magic. Each tool is explicitly bound to a Python function, then described for the model with @tool. The bound functions live in the companion module. 

@bind search_web language: python from: "./companions/recurring_topic_monitor.py" symbol: search_web
@bind search_news language: python from: "./companions/recurring_topic_monitor.py" symbol: search_news
@bind crawl_url language: python from: "./companions/recurring_topic_monitor.py" symbol: crawl_url

@tool search_news
  Search current news. Use for fresh announcements, reporting, changes,
  cancellations, and independent coverage.
  - query: string (required) - Focused news query.
  - max_results: integer - Maximum result count, normally 5 to 7.
  - time_range: string - Recency window: d, w, m, y, or any.

The companion is intentionally thin. It exposes exactly the three functions the promplet binds: 

async def search_news(query: str, max_results: int = 7, time_range: str = "w") -> str:
    """Execute one news search exactly as requested by the promplet."""

async def crawl_url(url: str) -> str:
    """Crawl one URL exactly as requested by the promplet."""
Why this matters: the promplet owns the reasoning and tool contract; the companion owns the actual side effect. That keeps the prompt readable and the executable surface reviewable.

3. Constrain the tool workflow

After declaring tools, the spec tells the model how to use them. The recurring monitor must search before synthesizing, crawl selected sources, cluster duplicate coverage, and classify findings as new, repeated, materially changed, or uncertain. 

## Required tool workflow

You MUST use the bound tools before writing the digest. Do not answer from model
memory.

1. Translate the monitor brief into focused query families.
2. Run several focused searches rather than one broad query.
3. Select the strongest primary sources and independent sources.
4. Crawl individual announcement, article, report, model-card, calendar, or
   booking pages.
5. Cluster duplicate coverage of the same underlying story/event.
6. If previous-report memory is present, classify each cluster:
   - new
   - repeated/no material change
   - material update
   - uncertain match

This is where executable prompting becomes more than "give the model a search tool." The spec defines a bounded investigation protocol and an output contract. 

@output enforce: strict
  Return Markdown with:
  1. Monitor snapshot
  2. Top findings
  3. Suppressed repeats
  4. Source map and gaps
  5. Watchlist for next run
  6. Action checklist

4. Run the checked-in monitor

The example runner points the executable promplet at a concrete variable file and saves both the digest and trace. The default input is an AI-news monitor. 

bash examples/saved-artifact-workflows/recurring-topic-monitor/run.sh

With its default input, the script expands its variables to this exact weavemark command: 

weavemark library builtin:catalog/executable/recurring-topic-monitor \
  --vars-file examples/saved-artifact-workflows/recurring-topic-monitor/inputs/ai-news.json \
  --run \
  --no-protections \
  --output examples/saved-artifact-workflows/recurring-topic-monitor/outputs/ai-news/execution-output.md \
  --show-output \
  --no-file-summary \
  --trace-output examples/saved-artifact-workflows/recurring-topic-monitor/outputs/ai-news/execution-trace.md \
  --verbose \
  --batch-only
About --no-protections: the checked-in example uses local Python bindings from this repository. Keep that trust decision explicit; do not use it for arbitrary companion code you have not reviewed.

The runner writes:

  • execution-output.md - the final monitor digest.
  • execution-trace.md - metadata and every tool call with arguments and returned content.

5. Adapt the same executable prompt

To monitor another recurring topic, keep the executable spec and change the variables. The second checked-in input switches from AI-news monitoring to family-friendly events in Sao Paulo. 

bash examples/saved-artifact-workflows/recurring-topic-monitor/run.sh \
  examples/saved-artifact-workflows/recurring-topic-monitor/inputs/child-events.json

The input file changes the topic, mode, lookback window, region, audience, and practical constraints: 

{
  "topic": "family-friendly children's events, workshops, exhibits, and performances",
  "monitor_mode": "events",
  "lookback_window": "next 30 days",
  "region": "Sao Paulo, Brazil",
  "audience": "parents planning weekend activities for young children"
}
What changed? The same bound tools and investigation protocol now search calendars and event pages instead of AI product news. The executable surface stays stable; the monitor brief changes.

Final step: Use the output

This tutorial output is already the executed monitor result. It is not merely a prompt to paste into a chat assistant.

What to do with the output: read execution-output.md as the finished recurring-topic digest, share it with the intended audience, or feed it into a bespoke downstream workflow that consumes Markdown reports. Use execution-trace.md to audit the exact tool calls, sources, failures, and evidence path. Only paste either file into ChatGPT, Claude, Gemini, or Copilot Chat if you want a separate critique, summary, or follow-up plan.

What you learned

  • @execute single-call lets a promplet run with bounded tool access.
  • @bind connects a prompt-visible tool name to a reviewed Python function.
  • @tool documents the callable surface the model may use.
  • The prompt can require a search/crawl workflow before synthesis.
  • Executable runs leave both a user-facing output and a trace for audit.
