Build a useful browser app from a concise promplet.

1. Keep the source focused on the product

The source promplet is intentionally under 120 lines. It names the product, board behavior, essential flows, experience, deliverable, and output shape. It does not repeat generic browser implementation advice: 

@promplet version: 0.7

@refine module:weavemark.domains.programming.foundations.base_spec_author
@refine module:weavemark.domains.programming.stacks.browser_static_esmodules
@refine module:weavemark.domains.programming.types.browser_file_backed_webapp
@refine module:weavemark.domains.programming.modules.browser_sqlite_file_store
@refine module:weavemark.domains.programming.modules.workflow_board
@refine module:weavemark.domains.programming.modules.card
@refine module:weavemark.domains.programming.modules.activity_stream
@refine module:weavemark.domains.programming.modules.output_surfaces
@refine module:weavemark.domains.programming.modules.browser_ai_handoff

# AI Kanban — Browser Workspace for Human-AI Work

The remaining source says what is distinctive: six ordered columns, cards as human-AI workspaces, transactional movement and activity, provider-neutral handoff packets, calm interaction, and a complete static deliverable.

The balance: keep choices that define this product in the entrypoint. Move stable architecture that another browser app could reuse into named modules.

2. Refine reusable architecture

Four general modules make this browser app possible without making the AI Kanban source verbose:

  • Static ES modules: relative assets, workers, accessibility, responsive behavior, and GitHub Pages delivery.
  • File-backed web app: open, create, reconnect, fallback import/download, dirty-state guards, and permission recovery.
  • Browser SQLite store: sql.js in a worker, migrations, transactions, export, conflict detection, and save serialization.
  • AI handoff: provider-neutral request and response packets with validation, preview, approval, and provenance.

Domain modules then contribute board, card, activity, and typed-output concepts. The entrypoint composes those capabilities and adds only the decisions that make them AI Kanban.

3. Compile the implementation contract

Compile the source to see the full engineering brief before an agent writes code: 

mkdir -p outputs/tutorial-ai-kanban
weavemark promplets/catalog/standalone/ai-kanban-board.weavemark.md \
  --batch-only \
  --output outputs/tutorial-ai-kanban/compiled-spec.md

The compiled document expands the concise source into seven exact sections covering architecture, SQLite operations, domain behavior, interaction states, recovery, file structure, and acceptance tests. It requires a worker-owned database, complete-file save semantics, explicit conflict handling, and honest fallback copy.

Inspect before building: compilation produces a plain Markdown artifact. Review it, revise the source or reusable modules if it is weak, and compile again. Do not rely on a programming agent to invent missing architecture silently.

4. Implement it explicitly

Hand the reviewed contract to a configured programming agent in a fresh workspace: 

weavemark implement outputs/tutorial-ai-kanban/compiled-spec.md \
  --name tutorial-ai-kanban \
  --profile copilot

Use --dry-run first if you want WeaveMark to prepare the workspace, manifest, implementation prompt, and resolved command without launching the agent.

The run records the compiled spec, implementation prompt, transcript, and manifest, making the run traceable. It also isolates generated work in a dedicated directory, but this is workspace separation, not an operating-system security sandbox. Review the profile and command before running code you do not trust.

5. Understand the local-file lifecycle

The generated app has no backend. SQLite runs inside a Web Worker, and the database file chosen by the user remains canonical:

  • Chromium desktop: Open or Create grants a file handle; connected changes autosave back to that file.
  • Other browsers: import an existing file and explicitly download each saved replacement.
  • IndexedDB: remembers only the recent file handle, never a shadow canonical database.
  • Conflicts: an external file change stops autosave and offers Reload, Save As, or explicit Overwrite.
  • Coordination: Web Locks selects one writer and BroadcastChannel reports saves from another tab.
Why the inefficiency is acceptable: sql.js imports and exports the complete database. Rewriting a whole file is a poor fit for a large database, but it is simple, inspectable, and practical for board-sized personal workspaces.

6. Run and verify the checked-in app

Serve the repository over HTTP because browser workers, WASM, and native file APIs do not behave correctly from a raw file: URL: 

python -m http.server 4173

open http://127.0.0.1:4173/outputs/implementations/ai-kanban-browser/

In another terminal, run the deterministic packet tests: 

cd outputs/implementations/ai-kanban-browser
npm test

The repository tests additionally verify the static asset contract, absence of backend runtime calls, SQLite schema, transactional repository operations, compiled obligations, Pages publication, and home hero.

7. Inspect the real result

The checked-in AI Kanban implementation demonstrates one successful end-to-end run of the loop. It is a real static application, not a mockup: cards, plans, outputs, activity, keyboard and pointer movement, file persistence, conflict recovery, and AI handoffs all work in the browser.

Final step: Use the output

Start with Try demo to explore without claiming durable storage. Then create a real .aikanban.sqlite workspace, add a card and plan, save it, close the workspace, and reopen the file. Export an AI handoff packet when you want help from any assistant; import the response and approve its proposed changes before they reach the board.

What to do with the output: use the live app or the checked-in outputs/implementations/ai-kanban-browser/ project as a working personal board, an implementation reference, or a starting point for another local-file browser application. Keep the compiled spec beside the project so future changes remain traceable to the intended contract.

What you learned

  • A concise entrypoint can remain product-specific while reusable modules carry implementation depth.
  • Compilation gives you a reviewable contract before implementation begins.
  • weavemark implement can hand that contract to a configured programming agent in a separate workspace.
  • A static browser app can use a user-selected SQLite file as canonical persistence without a backend.
  • Provider-neutral handoff keeps the useful core independent of any AI vendor.
  • The real app, tutorial, home hero, tests, and Pages demo now share the same architecture.
